### 创建节点收集依赖

packages\reactivity\src\ref.ts

```ts
...

class RefImpl {
  ...

  get value() {
    if (activeSub) {
      // 如果activeSub有，那就保存，等更新的时候触发
      const newLink: Link = {
        sub: activeSub,
        nextSub: undefined,
        prevSub: undefined,
      }

      /**
       * 把新的订阅者添加到链表的尾部
        1. 如果链表不为空，就把新的订阅者添加到链表的尾部
        2. 如果链表为空，就把新的订阅者作为链表的头节点和尾节点
       */
      if (this.subsTail) {
        this.subsTail.nextSub = newLink
        newLink.prevSub = this.subsTail
        this.subsTail = newLink
      } else {
        this.subs = newLink
        this.subsTail = newLink
      }
    }

    // 依赖收集
    return this._value
  }

  ...
}

...
```

getter 中会先判断当前是否存在依赖，如果存在的话创建一个节点。接着判断当前链表中是否存在节点，如果存在尾节点，表示链表中是存在节点的，那么把新创建的节点放到尾节点，并且移动尾节点的位置；如果不存在尾节点表示当前是不存在节点的（这里是双向链表），那就直接把头结点和尾结点都设置为新创建的节点（头尾相同）。

### 代码优化 system

（新增）packages\reactivity\src\system.ts

```ts
/**
 * 链表节点
 */
export interface Link {
  // 保存effect
  sub: Function

  // 下一个节点,如果没有就是undefined
  nextSub: Link | undefined

  // 上一个节点,如果没有就是undefined
  prevSub: Link | undefined
}

/**
 * 链接ref和effect，建立ref和effect之间的链表关系
 * @param dep ref对象
 * @param sub effect函数（依赖项）
 */
export function link(dep, sub) {
  const newLink: Link = {
    sub,
    nextSub: undefined,
    prevSub: undefined,
  }

  /**
   * 把新的订阅者添加到链表的尾部
   * 1. 如果链表不为空，就把新的订阅者添加到链表的尾部
   * 2. 如果链表为空，就把新的订阅者作为链表的头节点和尾节点
   */
  if (dep.subsTail) {
    dep.subsTail.nextSub = newLink
    newLink.prevSub = dep.subsTail
    dep.subsTail = newLink
  } else {
    dep.subs = newLink
    dep.subsTail = newLink
  }
}

/**
 * 传播更新
 * @param subs 依赖项
 */
export function propagate(subs) {
  let link = subs
  const queueEffect = []
  while (link) {
    queueEffect.push(link.sub)
    link = link.nextSub
  }
  queueEffect.forEach(effect => effect())
}
```

system.ts 文件主要保存的是收集依赖创建节点，触发更新

### 代码优化 ref

packages\reactivity\src\ref.ts

```ts
import { activeSub } from './effect'
import { link, Link, propagate } from './system'

enum ReactivityFlags {
  IS_REF = '__v_isRef',
}

class RefImpl {
  _value; // ref获取到的值（实际值）

  [ReactivityFlags.IS_REF] = true // 一个标记，表示这个对象是一个ref对象

  /**
   * 订阅者链表的头节点
   */
  subs: Link

  /**
   * 订阅者链表的尾节点
   */
  subsTail: Link

  constructor(value) {
    this._value = value
  }

  /**
   * get 用来收集依赖
   */
  get value() {
    if (activeSub) {
      trackRef(this)
    }

    return this._value
  }

  /**
   * set 用来收集更新
   */
  set value(newVal) {
    this._value = newVal
    triggerRef(this)
  }
}

export function ref(value) {
  return new RefImpl(value)
}

/**
 * 判断一个对象是不是一个ref对象
 * @param value 对象
 * @returns 返回布尔，true表示是一个ref对象，否则是其他
 */
export function isRef(value) {
  return !!(value && value[ReactivityFlags.IS_REF])
}

/**
 * 收集依赖，建立ref和effect之间的链表关系
 * @param dep ref对象
 * @return void
 */
export function trackRef(dep) {
  if (activeSub) {
    link(dep, activeSub)
  }
}

/**
 * 触发更新，遍历链表，执行所有的effect函数
 * @param dep ref对象
 * @return void
 */
export function triggerRef(dep) {
  if (dep.subs) {
    propagate(dep.subs)
  }
}
```

### 代码优化 effect

（新增）packages\reactivity\src\effect.ts

```ts
// 全局变量，用来保存当前正在执行依赖（effect的回调函数）
export let activeSub

export class ReactiveEffect {
  fn: Function

  constructor(fn) {
    this.fn = fn
  }

  run() {
    // 每次执行之前把this放到activeSub上，这样在依赖收集的时候就能知道当前正在执行的effect函数了
    activeSub = this
    try {
      return this.fn()
    } finally {
      // 执行完成之后把activeSub置空，这样在依赖收集的时候就知道没有正在执行的effect函数了
      activeSub = undefined
    }
  }
}

// 依赖收集和触发更新的函数
export function effect(fn) {
  const e = new ReactiveEffect(fn)
  e.run()
}
```

通过一个类来实现了触发 effect 的行为，之前是直接调用 activeSub 来触发依赖的更新，但是这里实现了一个类，通过类的 run 方法来触发依赖的更新

## effect 嵌套

目前的 effect 存在一个问题，嵌套问题

```ts
<!doctype html>
<html lang="en">
  <head>
    <meta charset="UTF-8" />
    <meta name="viewport" content="width=device-width, initial-scale=1.0" />
    <title>Document</title>
  </head>
  <body>
    <script type="module">
      import { ref, effect } from '../dist/reactivity.esm.js'

      const count = ref(0)
      effect(() => {
        effect(() => {
          console.log('effect run-嵌套的', count.value)
        })
        console.log('effect run-外部的', count.value)
      })

      setTimeout(() => {
        count.value = 1
      }, 1000)

```

上面嵌套了两个 effect，打印结果是这样

这里一秒钟之后打印了 1，但是正常来讲应该是两个 1 的，因为上面两个依赖。
原因也很简单，就是在这里：
![[4-响应式（2）-2AED96C026AEA5410FAF4B24734E8145.png]]
第一层的时候 activeSub 被赋值，第二次进来的时候 activeSub 有值了，但是会重新被赋值一遍（activeSub 是全局变量）,然后执行 this.fn() ，然后执行 activeSub = undefined.
接着会走外层的 `console.log('effect run- 外部的 ', count.value)` 这行代码，这里会走到 ref 里面的 getter

这样代码就执行完了。
接着执行 setTimeout 中把 count 设置为 1，执行
![[4-响应式（2）-69DA0AA16862F54C4A18A7896A24F4D3.png]]
接着 triggerRef 中会拿到这个依赖项收集到的所有依赖并执行
![[4-响应式（2）-65775182C8F87AC0CAF59F5AC0C79B46.png]]

说了这么多，简单来说就是外层 effect 没有收集到依赖，只收集到内层 effect 的依赖，所以在一秒钟之后才打印一次

### 保存上一次的 sub

packages\reactivity\src\effect.ts

```ts
// 全局变量，用来保存当前正在执行依赖（effect的回调函数）
export let activeSub

export class ReactiveEffect {

	...

  run() {
    // 执行run函数之前先把当前的sub保存起来
    const prevSub = activeSub;
    activeSub = this
    try {
      return this.fn()
    } finally {
      // 这里恢复之前的effect函数，这样在处理嵌套的逻辑的时候就能正确地收集依赖了
      activeSub = prevSub
    }
  }
}

	...

```

新增一个 prevSub 来记录 sub，在进入第二层 sub 的时候，通过 prevSub 来记录第一层的 sub，当第一层执行完了以后 activeSub 再赋值为 prevSub 就没问题了，执行 get 的时候 activeSub 就是有值的
打印：
![[4-响应式（2）-DB61F8D0586E577B7CDC429076EC43F8.png]]
第一次的时候执行两个 0，一秒钟之后执行了三个 1，为什么是三个？
收集：
代码进入第一层 effect 后，它是先执行第二层的 effect，然后执行第二层的 count，这里就**收集**第二层的依赖了，然后第二层的 effect 执行完了，执行第一层的 count 打印，这个时候，这里**收集**第一层的依赖。这里一共收集了两个依赖。

触发：
等到一秒钟之后，先**执行第二层的 sub**，打印嵌套，然后执行完以后，执行第一层的 sub，**执行第一层的 sub**，会先**执行第二层 sub**，打印嵌套，第二层执行完了以后，执行第一层的 count 打印。

## scheduler

### 触发 scheduler

什么是 scheduler?
scheduler 是 vue 里面的一个概念，翻译成中文是调度器的意思
在 vue 中，他的使用方法是

```ts
effect(
  () => {
    console.log('effect run-外部的', count.value)
  },
  {
    scheduler(fn) {
      console.log('执行了调度器')
    },
  },
)
setTimeout(() => {
  count.value = 1
}, 1000)
```

这部分代码的意思是：在 effect 的 sub 中收集依赖，当 setTimeout 执行的时候，触发更新，但是触发的却是 scheduler 的更新。
使用官方包后打印结果：
![[4-响应式（2）-A4112EC227AF2C12C2C29DECF44FA3E1.png]]
可以看出来，在更改了 count 后，触发的是调度器，而不是 sub
effect 函数会传入第二个参数，所以需要对 effect 函数做一个改造
packages\reactivity\src\effect.ts

```ts
...
// 依赖收集和触发更新的函数
export function effect(fn, options) {
    ...
}
```

上面对 effect 函数新增了一个 options 参数

packages\reactivity\src\effect.ts

```ts
export class ReactiveEffect {

    ...

  scheduler() {
    this.run()
  }
}
```

对 ReactiveEffect 类新增一个 scheduler 函数，这个函数直接调用 run 函数

它是定义在原型上的，意味着，如果实例有 scheduler，那么会把它（原型上的 scheduler）覆盖掉，执行覆盖的 scheduler，如果实例没有的话，就会执行原型上的 scheduler，也就是上面的 scheduler。

### 合并 scheduler

## effect 的返回

effect 是有返回值的，返回的是一个函数，在官方中是有的，在往下看之前可以打印一下官方的返回

```ts
// 依赖收集和触发更新的函数
export function effect(fn, options) {
  const e = new ReactiveEffect(fn)
  Object.assign(e, options)
  e.run()

  const runner = e.run.bind(e)

  /**
   * 把effect实例放到函数属性中
   */
  runner.effect = e
  return runner
}
```

这里的 effect 就是 sub 了，返回的是 run 函数

> 注意：这里不能直接返回 e.run 会丢失 this

## scheduler 中触发 effect 的 sub

当 scheduler 实现之后，是不是意味着我们不能调用 effect 依赖函数呢？
不是的

```html
<!doctype html>
<html lang="en">
  <head>
    <meta charset="UTF-8" />
    <meta name="viewport" content="width=device-width, initial-scale=1.0" />
    <title>Document</title>
  </head>
  <body>
    <script type="module">
      import { ref, effect } from '../dist/reactivity.esm.js'
      const count = ref(0)
      let runner = effect(
        () => {
          console.log('effect run-外部的', count.value)
          return 123
        },
        {
          scheduler() {
            const res = runner() // 这里调用的就是effect的第一个参数（上面的）
            console.log('触发了调度器', res) // 123
          },
        },
      )

      console.dir(runner)
      setTimeout(() => {
        count.value = 1
      }, 1000)
    </script>
  </body>
</html>
```

这里的 scheduler 中调用了 runner，这个 runner 实际上就是 effect 的第一个参数了，这个函数返回的是 123，所以 scheduler 里面的 res 就是 123 了
