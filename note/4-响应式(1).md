## 创建节点，收集依赖

packages\reactivity\src\ref.ts

```ts

class RefImpl {

	...

  get value() {
    if (activeSub) {
      // 创建链表节点，节点中的sub是依赖（订阅者）
      const newLink: Link = {
        sub: activeSub,
        nextSub: undefined,
        prevSub: undefined,
      }

      /**
       * 存在两种情况：
       * 1.链表为空，链表为空意味着尾节点为空，所以给头结点和尾结点赋值(头尾相同)
       * 2.链表不为空，链表不为空意味着尾节点有值，那么在尾节点的后面添加新的节点
       */
      if (this.subsTail) {
        this.subsTail.nextSub = newLink
        newLink.prevSub = this.subsTail
        this.subsTail = newLink
      } else {
        this.subs = newLink
        this.subsTail = newLink
      }
    }
    return this._value
  }

	...
}

	...
```

> 注意：
>
> 1. getter 中会先判断当前是否存在依赖，如果存在的话创建一个节点
> 2. 判断当前链表中是否存在尾节点，如果存在尾节点，表示链表中是存在节点的，那么把新创建的节点放到尾节点，并且移动尾节点的位置；
> 3. 如果不存在尾节点表示当前是不存在节点的（这里是双向链表），那就直接把头节点和尾节点都设置为新创建的节点（头尾相同）。

```ts
export interface Link {
	...
}

/**
 * 链接ref和effect，建立ref和effect之间的链表关系
 * @param dep ref对象
 * @param sub effect函数（依赖项）
 */
export function link(dep, sub) {
  // 创建链表节点，节点中的sub是依赖（订阅者）
  const newLink: Link = {
    sub: sub,
    nextSub: undefined,
    prevSub: undefined,
  }

  /**
   * 存在两种情况：
   * 1.链表为空，链表为空意味着尾节点为空，所以给头结点和尾结点赋值(头尾相同)
   * 2.链表不为空，链表不为空意味着尾节点有值，那么在尾节点的后面添加新的节点
   */
  if (dep.subsTail) {
    dep.subsTail.nextSub = newLink
    newLink.prevSub = dep.subsTail
    dep.subsTail = newLink
  } else {
    dep.subs = newLink
    dep.subsTail = newLink
  }
}

/**
 * 触发依赖
 * @param subs 链表节点
 */
export function propagate(subs) {
  let link = subs // 拿到头结点
  const queueEffect = [] // 保存所有依赖
  while (link) {
    queueEffect.push(link.sub) // 把节点中依赖保存到数组中
    link = link.nextSub // 节点赋值为下一个节点
  }

  // 触发依赖函数
  queueEffect.forEach(effect => effect())
}

```

system.ts 文件主要保存的是收集依赖创建节点，触发更新

```ts
  }
}
```

### effect

packages\reactivity\src\effect.ts

```ts
// 全局变量，用来保存当前正在执行的依赖（订阅者），这里需要导出。
export let activeSub
export class ReactiveEffect {
  fn: Function

  constructor(fn) {
    this.fn = fn
  }

  run() {
    //
    /**
     * 每次执行之前把 this 放到 activeSub 上,
     * 这样在依赖收集的时候就能知道当前正在执行的effect函数了
     */
    activeSub = this

    try {
      return this.fn()
    } finally {
      /**
       * 执行完成之后把activeSub置空，
       * 这样在依赖收集的时候就知道没有正在执行的effect函数了
       */
      activeSub = undefined
    }
  }
}

export function effect(fn) {
  const e = new ReactiveEffect(fn)
  e.run()
}
```

### 保存上一次的 sub

packages\reactivity\src\effect.ts

```ts
	...

export class ReactiveEffect {
	...

  run() {
    // 执行run函数之前先把当前的sub保存起来
    const prevSub = activeSub
    activeSub = this
    try {
      return this.fn()
    } finally {
      // 内层依赖执行完，如果外层有依赖，这里重新赋值为外层依赖
      activeSub = prevSub
    }
  }
}
	...
```

新增一个 prevSub 来记录 sub，在进入第二层 sub 的时候，通过 prevSub 来记录第一层的 sub，当第一层执行完了以后 activeSub 再赋值为 prevSub 就没问题了，执行 get 的时候 activeSub 就是有值的
打印：
![[4-响应式(1)-DB61F8D0586E577B7CDC429076EC43F8.png]]
第一次的时候执行两个 0，一秒钟之后执行了三个 1，为什么是三个？
收集：

## scheduler

### 触发 scheduler

什么是 scheduler?
scheduler 是 vue 里面的一个概念，翻译成中文是调度器的意思
在 vue 中，他的使用方法是

```ts
effect(
  () => {
    console.log('effect run-外部的', count.value)
  },
  {
    scheduler(fn) {
      console.log('执行了调度器')
    },
  },
)
setTimeout(() => {
  count.value = 1
}, 1000)
```

这部分代码的意思是：在 effect 的 sub 中收集依赖，当 setTimeout 执行的时候，触发更新，但是触发的却是 scheduler 的更新。
使用官方包后打印结果：
![[4-响应式(1)-A4112EC227AF2C12C2C29DECF44FA3E1.png]]
可以看出来，在更改了 count 后，触发的是调度器，而不是 sub
effect 函数会传入第二个参数，所以需要对 effect 函数做一个改造

packages\reactivity\src\effect.ts

```ts
	...

export function effect(fn, options) {
    ...

}
```

上面对 effect 函数新增了一个 options 参数

packages\reactivity\src\effect.ts

```ts
export class ReactiveEffect {

    ...

  scheduler() {
    this.run()
  }
}
```

对 ReactiveEffect 类新增一个 scheduler 函数，这个函数直接调用 run 函数

它是定义在原型上的，意味着，如果实例有 scheduler，那么会把它（原型上的 scheduler）覆盖掉，执行覆盖的 scheduler，如果实例没有的话，就会执行原型上的 scheduler，也就是上面的 scheduler。

## effect 的返回

```ts
export function effect(fn, options) {
  const e = new ReactiveEffect(fn)
  Object.assign(e, options)
  e.run()

  const runner = e.run.bind(e)

  // 把effect实例放到函数属性中
  runner.effect = e
  return runner
```

## scheduler 中触发 effect 的 sub

当 scheduler 实现之后，是不是意味着我们不能调用 effect 依赖函数呢？
不是的

```html
<!doctype html>
<html lang="en">
  <head>
    <meta charset="UTF-8" />
    <meta name="viewport" content="width=device-width, initial-scale=1.0" />
    <title>Document</title>
  </head>
  <body>
    <script type="module">
      import { ref, effect } from '../dist/reactivity.esm.js'
      const count = ref(0)
      let runner = effect(
        () => {
          console.log('effect run-外部的', count.value)
          return 123
        },
        {
          scheduler() {
            const res = runner() // 这里调用的就是effect的第一个参数（上面的）
            console.log('触发了调度器', res) // 123
          },
        },
      )

      console.dir(runner)
      setTimeout(() => {
        count.value = 1
      }, 1000)
    </script>
  </body>
</html>
```

这里的 scheduler 中调用了 runner，这个 runner 实际上就是 effect 的第一个参数了，这个函数返回的是 123，所以 scheduler 里面的 res 就是 123 了
